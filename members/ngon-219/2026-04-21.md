> Sử dụng Amazon EventBridge (CloudWatch Events) để gửi một event đến Amazon Kinesis Data Stream khi có file upload mới, sau đó dùng lambda function để consume và process data, lưu trữ dưới dạng Json trong Aurora Db cluster (sai)
>> Nguyên nhân: Kinesis data stream thường dùng là một data stream, được thiết kế để xử lý dữ liệu thời gian thực liên tục, và yêu cầu giữ nguyên thứ tự, tuy nhiên lại kém linh hoạt với tải đột biến. Mặc dù Aurora Db Cluster là một RDBMS nhưng lại hỗ trợ dữ liệu Json, nhưng dùng chỉ để lưu Json là thừa và đắt vc, dùng dynamo là hợp lí cho case này. > Dáp án đúng là dùng S3 để gửi event noti tới Amazon Simple Quêu Service queue. Dùn lamda function để đọc từ quêu và provcess data lưu Json file trong DynamoDB

# Câu 15: Một công ty có nhiều web server cần được truy cập thương xuyên vào một phiên bản Amazon RDS MySQL Multi-AZ chung. Công ty muốn phương thức bảo mật cho web server để kết nối db đồng thời đáp ứng yêu cầu bảo mật về việc thay đổi thông tin đăng nhập người dùng thường xuyên. 

> Suy nghĩ Dùng KMS Lưu trữ thông tin xác thực (credentials) của người dùng database trong các file được mã hóa bằng AWS Key Management Service (AWS KMS) trên hệ thống file (file system) của web server. Web server cần có khả năng giải mã các file này và truy cập vào database (sai)

>> Lí do sai: 
>>> Vì thay đổi mật khẩu thường xuyên, việc dùng KMS sẽ tạo ra gánh nặng vận hành, Khi đến hạn đổi mật khẩu, quy trình sẽ là: Đổi mật khẩu trong RDS -> Lấy mật khẩu mới mã hóa bằng KMS -> Tạo ra file mới -> Phân phối và ghi đè file này lên toàn bộ 50 cái web server kia. Việc này đòi hỏi bạn phải viết các script tự động hóa cực kỳ phức tạp, rất dễ xảy ra lỗi và rớt mạng

>>> Rủi ro bảo mật: Dù file đã được mã hóa bằng KMS, nhưng web server buộc phải có quyền (IAM Role) gọi KMS để giải mã. Nếu hacker chiếm được quyền điều khiển web server đó, chúng có thể tận dụng luôn IAM Role của server để yêu cầu KMS giải mã file và lấy được mật khẩu database một cách dễ dàng. Lưu credentials trực tiếp trên máy chủ EC2 (dù dưới hình thức nào) chưa bao giờ là Best Practice của AWS.

> Đáp án đúng: Lưu trữ thông tin xác thực (credentials) của người dùng database trong AWS Secrets Manager. Cấp các quyền IAM cần thiết để cho phép các web server truy cập vào AWS Secrets Manager.

>>> Tính năng "Sát thủ" - Automatic Rotation (Tự động xoay vòng): Đây là tính năng "ăn tiền" nhất của Secrets Manager và sinh ra chính xác để giải quyết yêu cầu của đề bài. Secrets Manager có khả năng tích hợp sẵn (native) với RDS. Bạn chỉ cần cấu hình: "Hãy đổi mật khẩu 30 ngày 1 lần". Đến hẹn, Secrets Manager sẽ tự động đàm phán với RDS để đổi mật khẩu mới một cách an toàn mà không cần con người can thiệp, không gây gián đoạn ứng dụng.

>>> Quản lý tập trung (Centralized Management): Các web server sẽ không lưu bất kỳ file nào chứa mật khẩu cả. Khi code ứng dụng chạy, nó sẽ gọi API của AWS (thông qua quyền IAM) lên Secrets Manager để hỏi: "Cho tôi xin mật khẩu mới nhất để vào DB". Secrets Manager sẽ trả về mật khẩu ở dạng text ngay trong bộ nhớ (RAM) của ứng dụng.

>>> Bảo mật tuyệt đối: Mật khẩu không bao giờ chạm vào ổ cứng (file system) của EC2. Khi server bị xóa đi (scale in), không có rủi ro lộ lọt thông tin.
